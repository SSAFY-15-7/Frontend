## 프레임워크 조합 — 왜 이걸 쓰나

| 선택 | 근거 |
|---|---|
| **Vue 3** | 선택지 없음 — SSAFY 공통 명세 기준. 기획서 6.1에도 "Vue 3 (명세 기준)" 명시 |
| **Vite** | Vue 3 공식 표준 빌드 도구. 프로젝트는 공식 스캐폴더(`npm create vue@latest`) 산출물 기반 |
| **TypeScript** | FE1/FE2 협업의 핵심이 스토어 인터페이스 계약(아래 협업 가이드 참조)인데, 계약을 타입으로 강제하기 위함 |
| **Pinia** | Vue 3 공식 상태관리. 소켓 이벤트(FE2)가 갱신하고 화면(FE1)이 구독하는 구조라 중앙 스토어가 필수 |
| **Tailwind CSS** | 디자인 토큰 전략 때문. "색·폰트는 토큰으로만, 하드코딩 hex 금지" 규칙을 유틸리티 클래스로 강제하기 좋음 |
| **axios** | 인터셉터(추후 JWT 첨부·401 처리)를 쓰기 위해 fetch 대신 채택 |

## 버전 — 왜 이 숫자인가

**개별적으로 고른 게 아니라, 세팅일(2026-07-20) 기준 공식 스캐폴더 + npm latest가 잡은 최신 안정판이다.**
`package.json`이 전부 `^`(caret) 범위인 이유. 팀원 간 실제 설치 버전은 커밋된 `package-lock.json`으로 고정된다 — **lock 파일 임의 삭제 금지, `npm install`로만 갱신.**

의미 있는 포인트만:

- **Tailwind v4 (v3 아님)** — v4는 `tailwind.config.js` 없이 CSS 안 `@theme` 블록으로 토큰을 정의한다.
  `src/app/styles/main.css`에 디자인 원본의 CSS 변수를 그대로 옮겨 적는 지금 구조가 v4라서 가능한 것.
  v3 문서/블로그를 참고할 때 주의 — 설정 방식이 완전히 다르다. 빌드 연동도 v4 전용 `@tailwindcss/vite` 플러그인 사용.
- **TypeScript만 `~`(tilde, 마이너 고정)** — `vue-tsc`가 TS 컴파일러 내부 API에 의존해서 TS가 올라가면 깨질 수 있다.
  create-vue 공식 템플릿의 의도적 제약이므로 **TS 버전을 임의로 올리지 말 것.**
- Vue 3.5 / Pinia 4 / vue-router 4 / Vite 8 — 서로 짝이 맞는 현재 메이저. 특별한 선택 아님.

## 구조 접근 방식 — A안(레이아웃 우선)

세팅 설계 시점에 세 가지 안을 비교했다:

- **A안 — 레이아웃 우선 (채택)**: 얇은 기반(스캐폴딩+토큰+라우터)을 먼저 깔고, 라이브 뷰 프레임을 조기에 만들어 FE2를 언블록. 공통 컴포넌트는 페이지를 만들면서 추출.
- B안 — 디자인 시스템 선구축: 공통 컴포넌트를 다 만들고 페이지 조립. → FE2가 라이브 프레임을 기다리게 됨 (기각)
- C안 — 페이지 병렬 돌격: 각자 페이지부터 구현. → 스타일 불일치·중복 컴포넌트 리스크 (기각)

`shared/ui/`가 비어 있는 건 이 때문이다 — **일부러 비워둔 것.** 같은 UI가 두 번째로 등장할 때 추출한다.
