# 역할 경계·인터페이스 계약 (FE1·FE2 합의안)

> 상태: **합의 완료 (2026-07-22)** — `AGENTS.md` 규칙 4, `02-협업-가이드.md`가 이 문서 기준으로 갱신됨.
> 배경: 기존 문서 간 모순 3개 — ① `features/live/` 폴더 소유권과 LiveLayout(FE1 산출물)의 충돌,
> ② live 스토어 계약에 state만 있고 action이 없음, ③ 대기방(FE1)의 유일한 활성 기능이 채팅(FE2)인 문제.

## 1. 폴더 소유권 (조정안)

```
features/
  live/
    LiveView.vue        FE1  라우트 진입 + 프레임 조립
    LiveLayout.vue      FE1  슬롯 프레임 (1주차 최우선 산출물)
    media/              FE2  WebRTC 스테이지, 스트림 바인딩
    socket/             FE2  STOMP 클라이언트, 이벤트 핸들러
  chat/                 FE2  ChatPanel — 대기방·라이브 공용
  bid/                  FE1  입찰 패널 (#action 슬롯)
  prediction/           FE1  예측 패널 (#action 슬롯)
  auth/ wallet/ room/ result/   FE1  (기존과 동일)
```

- 규칙 4 개정 문구(안): **FE2 소유 = `features/live/media·socket/`, `features/chat/`, `stores/live.ts`.**
  단 `features/live/LiveView.vue`·`LiveLayout.vue`는 FE1(프레임).
  상대 소유 컴포넌트는 **import는 자유, 내부 수정은 사전 합의**.
- 입찰·예측 패널을 `features/live/` 밖으로 빼는 이유: #action은 FE1 소유인데
  FE2 폴더 안에 두면 "폴더 = 소유권" 등식이 깨진다.

## 3. `stores/live.ts` 계약 초안 (커밋은 FE2)

FE1 화면(입찰·예측 패널, 타이머 칩)이 의존하는 시그니처. 이름·타입이 합의 대상이고,
구현은 1주차엔 가짜(setTimeout), 3주차에 실소켓으로 교체 — **시그니처는 그때도 불변**.

```ts
// state 추가분
bidStatus: 'idle' | 'pending' | 'accepted' | 'rejected'  // BID-06 내 입찰 결과
bidRejectReason: string | null
bidCount: number          // 비크리 공개 허용 범위 (건수만)
participantCount: number  // 〃 (참여자 수)

// actions — 구현 FE2, 호출 FE1
connect(roomId: number): void     // 입장 시 연결 + 구독 (대기방·라이브 공용)
disconnect(): void                // 퇴장 시 정리 (타이머·구독 해제 포함)
placeBid(amount: number): void    // 전송만. 결과는 bidStatus로 수신
sendChat(text: string, isQuestion?: boolean): void  // CHAT-06
```

- `placeBid`가 Promise를 반환하지 않는 이유: 소켓은 재연결·유실이 있어
  요청-응답으로 감싸면 깨지기 쉽다. 결과는 state(`bidStatus`)로 흐르고,
  FE1 패널은 watch로 접수/거절만 그린다. 목 구현도 같은 모양으로 흉내 가능.

## 4-1. 참여 역할 — 입찰과 예측은 배타적이다 (BID·PRED)

한 방에서 한 사람은 **입찰자(BIDDER) 또는 예측자(PREDICTOR) 중 하나로만** 참여한다.
DB 근거: `auction_participant`의 `UNIQUE (auction_room_id, member_id)` +
`CHECK (participation_type IN ('BIDDER','PREDICTOR'))`. `bid`·`prediction` 모두 이 참여 행을 FK로 참조한다.

**강제 선택은 없다.** 입장 시 역할을 고르게 하지 않는다 — 아무것도 하지 않고 구경만 해도 된다.
참여 행은 **첫 입찰 또는 첫 예측 시점에 생성**된다(BE 확인 완료). 따라서 상태는 둘뿐이다.

| 상태 | 참여 행 | 화면 |
|---|---|---|
| 참여 전 (구경) | 없음 | 입찰·예측 패널 **둘 다 사용 가능** |
| 참여 후 | 있음 (타입 확정) | 선택한 쪽만 사용 가능, 다른 쪽 **잠김** |

- **확정 이후 역할 변경은 불가하다** (기획·BE 확정). 변경 API는 제공되지 않는다.
  `bid`·`prediction`의 FK가 `ON DELETE RESTRICT`인 것과도 일치한다.
  따라서 FE는 역할 변경 UI를 만들지 않고, 되돌리기 흐름도 상정하지 않는다.
- **첫 액션에는 확인 단계를 둔다.** 되돌릴 수 없는 선택인데 버튼만 봐서는 알 수 없기 때문이다.
  예: "입찰에 참여하면 이 방에서는 예측 베팅에 참여할 수 없습니다. 계속하시겠어요?"
  두 번째 입찰부터는 띄우지 않는다(입찰은 `sequence_no`로 여러 번 가능).
  예측은 `prediction.auction_participant_id`가 UNIQUE라 애초에 1회뿐이다.
- **잠긴 쪽은 숨기지 않고 안내를 남긴다.** 패널이 통째로 사라지면 레이아웃이 흔들리고
  사용자가 기능이 없어진 것으로 오해한다. "입찰자로 참여 중입니다" 수준의 안내로 자리를 유지한다.

## 6. 소켓 이벤트 타입 초안 (백엔드 명세와 대조 필요)

모든 이벤트는 `{ type: string, data: object }` 래핑 (BE2·BE3 공통 규격).

| type | data | 비고 |
|---|---|---|
| `ENDS_AT_UPDATED` | `{ endsAt, extensionCount }` | BID-03 연장. **타이머 틱 이벤트는 없다** — 카운트다운은 `endsAt` 기준 클라이언트 렌더링 |
| `BID_PLACED` | `{ bidCount }` | 비크리 — 건수만. 금액·입찰자 없음 |
| `BID_RESULT` | `{ accepted, reason? }` | 본인 입찰 응답 → `bidStatus` 반영 |
| `CHAT_MESSAGE` | `{ id, nickname, text, isQuestion }` | |
| `VIBE` | `{ text }` | BID-04 |
| `PRESENCE` | `{ participantCount }` | |
| `AUCTION_CLOSED` | `{ resultId }` | 결과 화면 진입 트리거 |

## 7. 목(mock) 범위 — "목 소켓"이 아니라 핸들러 드라이버

- 가짜 STOMP 클라이언트(토픽 구독·연결 상태 흉내)는 **만들지 않는다.**
- FE2가 만드는 것: ① 이벤트 핸들러(`{type,data}` → 스토어 갱신, **실소켓에서 그대로 재사용**),
  ② 그걸 두드리는 `setInterval` 드라이버(`src/mocks/liveFeed.ts`, ~50줄),
  ③ 스토어 fixture·가짜 action.
- FE1 화면 개발에는 fixture + 가짜 action이면 충분 — 목 피드는 데모(중간 발표)용.

## 8. 1~2주차 착수 순서

| 주차 | FE1 | FE2 |
|---|---|---|
| 1 | 디자인 토큰 확정 → **LiveLayout 프레임** → 방 목록 | 이벤트 핸들러 + 목 드라이버 + 스토어 계약 커밋(§3) |
| 2 | 방 개설 폼 → 입찰 패널 → 예측 패널 | ChatPanel → 대기방 → WebRTC 스파이크 |
| 3~ | 지갑·마이페이지·결과, REST 연동 | 실 STOMP 연결(핸들러 재사용), WebRTC 본작업 |

3주차 FE2는 미디어 연동이 최대 리스크 구간이므로 다른 짐을 얹지 않는다.
