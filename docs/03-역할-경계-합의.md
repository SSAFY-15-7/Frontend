## 3. `stores/live.ts` 계약 초안 (커밋은 FE2)

FE1 화면(입찰·예측 패널, 타이머 칩)이 의존하는 시그니처. 이름·타입이 합의 대상이고,

```ts
// state 추가분
bidStatus: 'idle' | 'pending' | 'accepted' | 'rejected'  // BID-06 내 입찰 결과
bidRejectReason: string | null
bidCount: number          // 비크리 공개 허용 범위 (건수만)
participantCount: number  // 〃 (참여자 수)

// actions — 구현 FE2, 호출 FE1
connect(roomId: number): void     // 입장 시 연결 + 구독 (대기방·라이브 공용)
disconnect(): void                // 퇴장 시 정리 (타이머·구독 해제 포함)

```

- `placeBid`가 Promise를 반환하지 않는 이유: 소켓은 재연결·유실이 있어

  FE1 패널은 watch로 접수/거절만 그린다. 목 구현도 같은 모양으로 흉내 가능.

## 4-1. 참여 역할 — 입찰과 예측은 배타적이다 (BID·PRED)

한 방에서 한 사람은 **입찰자(BIDDER) 또는 예측자(PREDICTOR) 중 하나로만** 참여한다.
DB 근거: `auction_participant`의 `UNIQUE (auction_room_id, member_id)` +
`CHECK (participation_type IN ('BIDDER','PREDICTOR'))`. `bid`·`prediction` 모두 이 참여 행을 FK로 참조한다.

**강제 선택은 없다.** 입장 시 역할을 고르게 하지 않는다 — 아무것도 하지 않고 구경만 해도 된다.
참여 행은 **첫 입찰 또는 첫 예측 시점에 생성**된다(BE 확인 완료). 따라서 상태는 둘뿐이다.

| 상태 | 참여 행 | 화면 |
|---|---|---|
| 참여 전 (구경) | 없음 | 입찰·예측 패널 **둘 다 사용 가능** |
| 참여 후 | 있음 (타입 확정) | 선택한 쪽만 사용 가능, 다른 쪽 **잠김** |

- **확정 이후 역할 변경은 불가하다** (기획·BE 확정). 변경 API는 제공되지 않는다.
  `bid`·`prediction`의 FK가 `ON DELETE RESTRICT`인 것과도 일치한다.
  따라서 FE는 역할 변경 UI를 만들지 않고, 되돌리기 흐름도 상정하지 않는다.
- **첫 액션에는 확인 단계를 둔다.** 되돌릴 수 없는 선택인데 버튼만 봐서는 알 수 없기 때문이다.
  예: "입찰에 참여하면 이 방에서는 예측 베팅에 참여할 수 없습니다. 계속하시겠어요?"
  두 번째 입찰부터는 띄우지 않는다(입찰은 `sequence_no`로 여러 번 가능).
  예측은 `prediction.auction_participant_id`가 UNIQUE라 애초에 1회뿐이다.
- **잠긴 쪽은 숨기지 않고 안내를 남긴다.** 패널이 통째로 사라지면 레이아웃이 흔들리고
  사용자가 기능이 없어진 것으로 오해한다. "입찰자로 참여 중입니다" 수준의 안내로 자리를 유지한다.

**모드 전환 UI 자체는 있다 (와이어프레임 대조, 2026-07-22).** `live` 화면의 레일 상단에
`상향 입찰` / `예측 베팅` 전환 버튼이 실재하고, 반대편 독은 사라지는 대신 잠금 문구로 덮인다:
"🔒 이 방에 이미 입찰했어요. 입찰과 예측 베팅은 한쪽만 참여할 수 있어요 (상호 배타 — 2등가 조작 방지)".
즉 **보는 것은 자유롭게 오갈 수 있고, 잠기는 것은 참여(제출)뿐**이다.
"토글을 만들지 않는다"가 아니라 "토글이 역할 변경 요청이 되지 않는다"로 읽어야 한다.

## 6. 소켓 이벤트 계약 (백엔드 명세 기준, 2026-07-23 재작성)

> **출처: 백엔드 「최종(종합) 명세서」의 API 명세 테이블·WebSocket 이벤트 명세.**
> 기존 §6은 대조 전 초안이라 이벤트 이름이 대부분 실제와 달랐다 — 마이그레이션은 §6-7 참조.
> 이름·`data` 필드는 **백엔드 명세가 단일 소스**이며 이 표는 그 사본이다. 임의로 추가·개명하지 않는다.

모든 이벤트는 `{ type: string, data: object }` 래핑 (백엔드 공통 규격, AGENTS.md 규칙 9).

- `destination`은 메시지 본문에 넣지 않는다.
- `auctionId`는 구독 destination에 이미 있으므로 본문에서 제외한다.
- `serverTime`은 공통 필드가 아니라 `TIME_SYNC`의 `data`에만 들어간다.
- `eventId`·`sequence`는 초기 구현에서 쓰지 않는다(중복·유실 복구가 필요해지면 추가).

### 6-1. 연결·인증 (WS-01)

```
CONNECT /ws
accept-version:1.1,1.2
Authorization:Bearer {accessToken}
heart-beat:10000,10000
```

- 서버가 토큰을 검증해 세션 `Principal(memberId)`을 확정한다. **SEND·SUBSCRIBE의 회원 식별은 항상 세션 Principal 기준**이라 클라이언트가 memberId를 실어 보내지 않는다.
- 토큰 만료 시 **수신은 grace 기간 동안 유지되고 발신만 `AUTH_EXPIRED`로 차단**된다 → `/app/auth/renew`로 재인증.
- 실패 코드: `AUTH_INVALID`(서명·회원 상태), `AUTH_EXPIRED`(연결 시점 만료).

### 6-2. 메인 토픽 — `/topic/auctions/{auctionId}`

| type | data | 기능 ID | 비고 |
|---|---|---|---|
| `TIME_SYNC` | `serverTime, endAt` | BID-05 | 최초 입장·재연결·연장·주기 보정. **`serverOffset`은 여기서 얻는다** |
| `AUCTION_STARTED` | `startedAt, endAt` | ROOM-05 | 판매자 수동 시작 → **대기방 전원 라이브 전환 트리거** |
| `BID_ACCEPTED` | `bidCount, participantCount, vibeMessage` | BID-01·02·04 | 비크리 — 금액·입찰자·순위 없음. **Vibe는 별도 이벤트가 아니라 이 필드** |
| `AUCTION_EXTENDED` | `endAt, extensionCount, remainingExtensions` | BID-03 | +5분 연장. **타이머 틱 이벤트는 없다** — 카운트다운은 `endAt` 기준 클라이언트 렌더링 |
| `AUCTION_ENDED` | `endedAt, status` | BID-07 | 마감. 신규 입찰·예측 차단 + 집계 시작 |
| `AUCTION_SETTLED` | `winner, highestBidAmount, secondHighestBidAmount, finalPrice, resultType` | BID-07 | 낙찰·정산 완료. 전원 동시 공개 |
| `AUCTION_NO_SALE` | `reason, endedAt` | BID-07 | 유찰 + 홀드 반환 안내 |
| `PREDICTION_SETTLED` | `winningBucket, totalPool` | PRED-03 | 정산 완료 **신호만**. 분포는 PRED-02 REST로 조회 ⚠️ 필드명 재정의 예정(§6-8) |
| `AUCTION_PAUSED` | `pausedAt, graceEndAt, reason` | BID-05 | 판매자 연결 끊김 (상태: 예정) |
| `AUCTION_RESUMED` | `resumedAt, endAt` | BID-05 | 유예 내 재접속 (상태: 예정) |
| `AUCTION_CANCELED` | `reason, endedAt, resultPath` | BID-07·DEAL-09 | 강제 종료 + 환급 완료 (상태: 예정) |

BID-10(집계 중 `SETTLING` 상태 브로드캐스트)도 이 토픽으로 오지만 **이벤트 타입명이 명세에 없다** → §6-8.

### 6-3. 그 외 destination

| destination | 이벤트 | 기능 ID |
|---|---|---|
| `/topic/auctions/{id}/chat` | `CHAT_MESSAGE`, `NOTICE_*` | CHAT-01·02 |
| `/topic/auctions/{id}/questions` | `QUESTION_ADDED`, `QUESTION_REMOVED` | CHAT-06 |
| `/topic/auctions/{id}/viewers` | `VIEWER_COUNT` | LIVE-03 |
| `/user/queue/errors` | SEND·SUBSCRIBE 처리 실패 회신 | WS-04 |
| `/user/queue/moderation` | `MODERATION_WARNING`, `CHAT_MUTED` | AI-05 |

`VIEWER_COUNT`는 **고유 사용자 수가 아니라 활성 STOMP 세션 수**다(멀티탭이면 중복 집계).

### 6-4. 클라이언트 → 서버 (SEND)

| destination | 용도 | 기능 ID |
|---|---|---|
| `/app/auctions/{id}/chat` | 채팅 전송. `isQuestion=true`면 질문 큐로 | CHAT-01·06 |
| `/app/auth/renew` | 토큰 만료 후 재인증 | WS-01 |

채팅은 `clientMessageId`로 멱등 처리한다(재전송 시 중복 저장 방지).

**입찰·예측은 소켓으로 보내지 않는다.** 둘 다 REST다 — `POST /auctions/{id}/bids`(BID-06, `Idempotency-Key` 헤더), `POST /auctions/{id}/predictions`(PRED-01, `clientRequestId`). 개인 결과는 REST 응답으로 즉시 돌아오고, 소켓으로는 집계값만 브로드캐스트된다.

### 6-5. 재접속·스냅샷 (WS-03)

재접속 시 이벤트 누락·역전을 막기 위해 **서버 권위 스냅샷 + 단조 증가 `stateVersion`**을 쓴다.

```
GET /api/v1/auctions/{auctionId}/realtime-state
→ { stateVersion, auctionStatus, endAt, remainingSeconds, extendCount,
    bidCount, bidParticipantCount, viewerCount,
    myBidAmount, myParticipationType, pinnedNotice, amIMuted }
```

절차:

1. SUBSCRIBE에 `receipt`를 넣고, RECEIPT 수신 전 이벤트는 **버퍼링**한다.
2. 스냅샷을 조회한다.
3. 버퍼에서 `event.stateVersion <= snapshot.stateVersion`인 것은 폐기한다.
4. 더 큰 버전만 오름차순 적용하고, 이후 실시간 이벤트를 처리한다.
5. 버전이 불연속·역전이면 **추측 적용하지 말고 스냅샷을 다시 조회**한다.

- `myBidAmount`는 본인 값만, 미입찰이면 `null`.
- `myParticipationType`은 `NONE | BID | PREDICTION` — **§4-1 배타 잠금 상태를 재접속 후 복원하는 근거값**이다.
- `bidParticipantCount`는 고유 회원 수(상향 입찰 중복 제거).

### 6-6. 목 드라이버가 지켜야 할 것

§7의 목 드라이버도 **위 이벤트 이름·필드를 그대로** 쓴다. 실소켓 전환 시 핸들러를 그대로 재사용하는 것이 §7의 전제이므로, 목에서만 통하는 이름을 만들면 그 전제가 깨진다.

### 6-7. 기존 초안 → 실제 명세 마이그레이션

핸들러가 초안 기준으로 작성됐다면 아래대로 교체한다.

| 기존 초안 | 실제 | 조치 |
|---|---|---|
| `ENDS_AT_UPDATED` | `AUCTION_EXTENDED` | 개명 + `remainingExtensions` 추가, `endsAt`→`endAt` |
| `BID_PLACED` | `BID_ACCEPTED` | 개명 + `participantCount`·`vibeMessage` 추가 |
| `BID_RESULT` | **없음** | 삭제 — 개인 결과는 BID-06 REST 201 응답 |
| `VIBE` | **없음** | 삭제 — `BID_ACCEPTED.vibeMessage`로 통합 |
| `PRESENCE` | `VIEWER_COUNT` | 개명 + destination 분리(`/viewers`) |
| `AUCTION_CLOSED` | `AUCTION_ENDED` → `AUCTION_SETTLED`/`AUCTION_NO_SALE` | 1개를 2~3단계로 분리. `resultId` 없음 |
| `CHAT_MESSAGE` | `CHAT_MESSAGE` | 유지하되 destination이 `/chat`으로 분리 |
| — | `TIME_SYNC`, `AUCTION_STARTED`, `AUCTION_PAUSED`/`RESUMED`/`CANCELED`, `PREDICTION_SETTLED` | 신규 추가 |

### 6-8. 백엔드에 확인이 필요한 항목

- **BID-10 집계 중 상태의 이벤트 타입명.** 명세에 destination과 설명만 있고 `type` 문자열이 없다.
- **`PREDICTION_SETTLED`의 `data` 필드명.** `winningBucket`·`totalPool`은 폐기된 구간(bucket) 설계의 잔재다. 신 설계(오차 밴드 ×5/×3/×1)에 맞는 이름으로 재정의가 필요하다. (`PREDICTION_UPDATED`는 2026-07-23 명세에서 삭제됨 — 실시간 분포는 신 설계에 존재하지 않는다.)
- **상태 변경 이벤트의 `stateVersion` 포함 여부.** WS-03은 "상태 변경 이벤트는 `stateVersion`을 포함해야 하며 BE3 이벤트 계약과 함께 확정한다"고만 적혀 있고 §6-2 표의 `data`에는 아직 없다.

### 6-9. §3 계약에 미치는 영향 (FE1·FE2 재논의 필요)

§3은 `bidStatus`/`bidRejectReason`을 두고 **"입찰 결과는 소켓(`BID_RESULT`)으로 흐른다"**는 전제 위에 설계됐다.
그런데 명세상 **`BID_RESULT` 이벤트는 없고, BID-06이 REST 201로 개인 결과를 동기 반환**한다:

```json
{ "myBidAmount": 82000, "previousBidAmount": 70000, "additionalHoldAmount": 12000,
  "totalHeldAmount": 82000, "availableBalance": 118000, "submittedAt": "..." }
```

거절도 HTTP 에러로 온다(`BID_BELOW_RESERVE_PRICE`, `BID_NOT_HIGHER_THAN_PREVIOUS`, `INSUFFICIENT_AVAILABLE_BALANCE`, `PREDICTION_ALREADY_JOINED` 등).
따라서 "`placeBid`가 Promise를 반환하지 않는다"는 §3의 근거가 성립하지 않는다.

§3은 명시적 합의 항목이므로 **이 갱신에서는 고치지 않고 논점만 남긴다.** 결정 후 별도 커밋으로 반영한다.

## 7. 목(mock) 범위 — "목 소켓"이 아니라 핸들러 드라이버

- 가짜 STOMP 클라이언트(토픽 구독·연결 상태 흉내)는 **만들지 않는다.**
- FE2가 만드는 것: ① 이벤트 핸들러(`{type,data}` → 스토어 갱신, **실소켓에서 그대로 재사용**),
  ② 그걸 두드리는 `setInterval` 드라이버(`src/mocks/liveFeed.ts`, ~50줄),
  ③ 스토어 fixture·가짜 action.
- FE1 화면 개발에는 fixture + 가짜 action이면 충분 — 목 피드는 데모(중간 발표)용.

## 8. 1~2주차 착수 순서

| 주차 | FE1 | FE2 |
|---|---|---|
| 1 | 디자인 토큰 확정 → **LiveLayout 프레임** → 방 목록 | 이벤트 핸들러 + 목 드라이버 + 스토어 계약 커밋(§3) |
| 2 | 방 개설 폼 → 입찰 패널 → 예측 패널 | ChatPanel → 대기방 → WebRTC 스파이크 |
| 3~ | 지갑·마이페이지·결과, REST 연동 | 실 STOMP 연결(핸들러 재사용), WebRTC 본작업 |

3주차 FE2는 미디어 연동이 최대 리스크 구간이므로 다른 짐을 얹지 않는다.
