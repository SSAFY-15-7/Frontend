### 라우트 설계 판단

**`/`는 세션에 따라 두 화면이 된다.** 와이어프레임의 `go('main')`은 `data-auth="out"`이면 `landing`으로 바꿔 보낸다.
경로를 나누지 않고 `/` 하나에서 세션 유무로 `LandingView`/`MainView`를 갈아 끼운다.
`/landing`을 따로 두면 로그인한 사용자가 그 링크를 받았을 때 갈 곳이 없고,
비로그인 사용자에게 `/`가 소개 페이지가 되는 게 와이어프레임의 동작이다.

**`main`과 `livelist`는 다른 화면이다.** 홈(`main`)은 캐러셀·마감 임박 중심의 큐레이션이고,
방 목록(`livelist`)은 검색·필터·무한 스크롤 중심의 탐색이다. 홈의 "전체 보기 →"가 목록으로 간다.
기존 표가 둘을 `/` 한 줄로 합쳐 놓았던 것이 오류다.

**방 개설 2단계는 라우트가 아니라 쿼리다.** `create`·`create2`는 별도 `data-screen`이지만
1단계 입력이 메모리에만 있어 2단계 URL을 새로고침하거나 공유해도 복원되지 않는다.
라우트는 `/rooms/new` 하나로 두고 단계는 `?step=1|2`로 표시한다 — 브라우저 뒤로가기가
"← 이전"과 같은 동작이 되고, 새로고침 시 스토어가 비어 있으면 1단계로 되돌린다.

**콘솔은 별도 라우트다.** 판매자와 구매자는 같은 방에서 완전히 다른 화면을 본다(`live` vs `console`).
`/rooms/:id/live` 안에서 역할 분기하면 한 컴포넌트가 두 화면을 떠안는다.
와이어프레임 헤더 내비게이션에도 "진행 콘솔"이 독립 항목으로 있다.

**유찰은 결과와 별도 라우트다.** `nosale`은 낙찰자·미낙찰자가 존재하지 않아 역할 분기 자체가 없고,
정산 카드 구성도 `result`와 다르다. 방 종료 후 진입 시 서버 결과에 따라
`/rooms/:id/result` ↔ `/rooms/:id/nosale`로 리다이렉트한다.
`docs/04` §3.3의 `result_status`가 `SOLD`/`NO_SALE`/`CANCELLED` 3값이므로
`CANCELLED`를 `nosale`의 어느 사유에 매핑할지는 **확인 필요**다
(와이어프레임 사유는 `nobid`·`left`·`forced` 3종이고, `forced`가 운영 강제 종료다).

**충전·출금은 라우트이고 직전 화면으로 복귀한다.** 와이어프레임에서 이 둘은 헤더 지갑 칩·마이페이지 지갑 탭·
라이브 시청 중 어디서든 열리고, 완료·취소하면 **들어오기 직전 화면으로 돌아간다**
(스크립트의 `chargeReturnTo`). 라우트로 두되 진입 직전 경로를 `route.query.from` 또는 스토어에 기억해 복귀시킨다.

### 화면 내 상태 (라우트가 아님)

| 화면 | 상태 | 값 | 분기 소스 |
|---|---|---|---|
| `result` | `data-role` | `win` / `lose` / `bet` / `none` | **스토어(서버 판정)** — 쿼리 금지 |
| `console` | `data-state` | `wait` / `live` | **스토어(방 상태)** |
| `nosale` | `data-why` | `nobid` / `left` / `forced` | **스토어(서버 결과)** |
| `ledger` | `data-cur` | `trade` / `bet` | 쿼리 `?cur=` |
| `mypage` | `vtabs` | `wallet` / `todo` / `deals` / `board` | 쿼리 `?tab=` |
| `mypage` › 경매 목록 | `stabs` | `won` / `host` / `join` | 컴포넌트 로컬 상태 |
| `livelist` | `stabs` + 칩 + 검색 | `live`/`wait`/`done`, 카테고리, 검색어 | 쿼리 `?state=`·`?cat=`·`?q=` |
| `live` | 독 모드 | 입찰 / 예측 (배타, `03-역할-경계-합의.md` §4-1) | 컴포넌트 로컬 + `live` 스토어 |

판단 기준은 **그 링크를 받은 사람이 같은 화면을 봐야 하는가**다.
`result`의 역할, `nosale`의 사유, `console`의 상태는 **서버가 판정하는 사실**이라 URL로 지정할 수 있으면 안 된다
(관전자가 `?role=win`으로 낙찰자 화면을 여는 일이 없어야 한다).
반대로 원장 통화·마이페이지 탭·목록 필터는 "여기 이 탭을 보라"고 링크를 보내는 게 자연스러워 쿼리로 둔다.
경매 목록 안의 하위 탭(`won`/`host`/`join`)은 이미 `?tab=deals`로 들어온 뒤의 2차 분류라 URL까지 끌고 가지 않는다.

와이어프레임 검증 결과 `console`의 `data-state`는 **`wait`·`live` 2종뿐이고 `done`은 없다.**
방송을 종료하면 콘솔에 머무르지 않고 낙찰 공개 연출을 거쳐 `result`/`nosale`로 이동한다.
판매자 전용 종료 후 화면이 따로 필요한지는 **확인 필요** — 현재는 판매자도 `result`를 본다는 전제다.

### `console` 담당 — FE1 확정 (2026-07-22)

콘솔의 구성 요소는 넷이다. ① 입찰 건수·참여자·남은 시간 지표 카드(대기 상태에서는 대기 인원·알림 신청·시작까지),
② 연장(최대 3회)·지금 시작·방송 종료·방 수정/삭제 제어, ③ 미답변 질문 큐, ④ 송출 미리보기 + 카메라·마이크 토글.

①②③은 FE1이 이미 맡은 방 관리·입찰 패널·결과 리포트와 같은 종류의 제어·통계 UI다.
FE2 고유 기술이 필요한 부분은 ④의 **로컬 송출 프리뷰**뿐이다.
그래서 **화면 전체는 FE1이 만들고, 송출 프리뷰 블록만 FE2가 `features/live/media/`에 컴포넌트로 제공해
FE1이 꽂는 형태**로 확정했다. LiveLayout `#video` 슬롯과 같은 패턴이라 새 계약이 필요 없다.
FE2의 3주차가 WebRTC 본작업으로 이미 가득 차 있다는 점(`03-역할-경계-합의.md` §8)도 근거다.

## 화면 이동 흐름

와이어프레임의 `data-go` 속성과 스크립트의 `go()` 호출을 전수 조사한 결과다.

**비로그인 진입 → 가입 → 온보딩**

| 출발 | 동선 | 도착 |
|---|---|---|
| `/` | 비로그인이면 홈 대신 랜딩 | `landing` |
| `landing` | "3초 만에 시작하기 →" · 하단 카카오 CTA | `signup` |
| `landing` | "라이브 구경하기" | `livelist` |
| `landing` | "이미 계정이 있다면 로그인" | `login` |
| `login` ↔ `signup` | 서로 상대 화면 링크 — 신규·기존을 막지 않는다 | — |
| `login`/`signup` | 카카오 인증 — **첫 로그인**(닉네임 없음 또는 필수 약관 미동의) | `onboard` |
| `login`/`signup` | 카카오 인증 — **기존 회원** | `main` |
| `onboard` | 닉네임 + 필수 약관 동의 후 "BidLive 시작하기" | `main` |

비로그인 상태에서는 헤더의 방 개설·진행 콘솔·마이페이지 링크가 숨겨진다(와이어프레임 CSS `body[data-auth="out"]`).
라우터 가드로 옮길 때도 같은 범위를 막는다.

**구매자 동선**

| 출발 | 동선 | 도착 |
|---|---|---|
| `main` | 캐러셀 카드 · 마감 임박 카드 | `live` |
| `main` | "전체 보기 →" · 헤더 "라이브" | `livelist` |
| `livelist` | 방 카드 — 방 상태 `live` | `live` |
| `livelist` | 방 카드 — 방 상태 `wait` | `wait` |
| `livelist` | 방 카드 — 방 상태 `done` | `result` (관전 역할) |
| `wait` | 판매자가 방송 시작 → 자동 전환 | `live` |
| `wait`/`live` | 뒤로가기 버튼 · "목록으로" | `livelist` |
| `live` | 하단 "다른 라이브" 카드 | `live` (다른 방) |
| `live` | 마감 → 집계 연출 → "낙찰 결과 자세히 →" | `result` |
| `live` | 마감 → 집계 연출 → "유찰 안내 자세히 →" | `nosale` |
| `live`/`wait` | 신고 누적 강제 종료 오버레이 → "정산 내역 보기" | `nosale` (`why=forced`) |
| `result` | "다른 방송 보러 가기 →" | `livelist` |
| `result` (낙찰자) | "배송 정보 입력하기" | 배송지 모달 (화면 이동 없음) |

강제 종료는 화면을 갈아끼우지 않고 보던 방송 위를 덮는 오버레이다 — `live`·`wait` 양쪽에 있고,
대기방에서는 잠겨 있던 입찰이 없으므로 "해제할 홀드가 없다"로 문구가 갈린다.

**판매자 동선**

| 출발 | 동선 | 도착 |
|---|---|---|
| `main` · 헤더 | "＋ 경매 방 개설" | `create` |
| `create` | "다음 · 경매 조건 →" | `create2` |
| `create` | "취소" | `main` |
| `create2` | "← 이전" | `create` |
| `create2` | 시작 방식 **지금 바로 시작** → "방송 시작하기" | `console` (`state=live`) |
| `create2` | 시작 방식 **예정 시각 지정** → 개설 | `console` (`state=wait`) |
| `console` (`wait`) | "지금 시작하기" | `console` (`state=live`) |
| `console` (`wait`) | "방 수정" | `create` |
| `console` (`wait`) | "방 삭제" / "방 삭제하기" | `main` |
| `console` (`live`) | "＋ 방송 5분 연장" (잔여 3회) | 제자리 갱신 |
| `console` (`live`) | "방송 종료" → 낙찰 공개 연출 | `result` / `nosale` |
| `nosale` | "이 상품 다시 등록하기" (판매자 전용, 강제 종료 시 숨김) | `create` |

와이어프레임의 `create2` "방송 시작하기"는 시연 편의상 구매자 `live`로 가지만,
개설자는 판매자이므로 실제 구현에서는 `console`로 보낸다(위 표는 실제 동선 기준).

**지갑·마이페이지 동선**

| 출발 | 동선 | 도착 |
|---|---|---|
| 헤더 아바타 | 마이페이지 | `mypage` (기본 탭 `wallet`) |
| 헤더 지갑 칩 | 충전 | `charge` |
| `mypage` › 지갑 탭 | 거래 포인트 카드 "거래 내역 보기 →" | `ledger` (`cur=trade`) |
| `mypage` › 지갑 탭 | 베팅 포인트 카드 "베팅 내역 보기 →" | `ledger` (`cur=bet`) |
| `mypage` › 지갑 탭 | 거래 카드의 "충전" / "출금" 버튼 | `charge` / `withdraw` |
| `ledger` | "← 마이페이지" | `mypage` |
| `charge` / `withdraw` | 완료 또는 취소 | **진입 직전 화면** (없으면 `mypage`) |
| `mypage` | "로그아웃" | `login` |
| `main` / `livelist` 티커 | 예측 수익률 스트립 클릭 | `mypage` (`tab=board`) |
| `mypage` › 지금 할 일 | "배송지 입력" | 배송지 모달 |
| `mypage` › 지금 할 일 | "발송했어요" / "받았어요" | 제자리 상태 전이 |

마이페이지 기본 탭은 `wallet`이다 — 지갑이 비면 아무것도 할 수 없는 구조라 진입 즉시 잔액을 보여준다.
"지금 할 일" 탭에는 **내가 움직여야 하는 건만** 올라가고 개수 배지를 상시 노출한다(DEAL-07).

## 현재 구현과의 차이

`src/app/router.ts`(9개 라우트, `dev` 기준) 대비. **이 문서는 계획이고, 코드 변경은 별도 PR에서 한다.**

| 현재 | 와이어프레임 기준 | 조치 |
|---|---|---|
| `/` = `RoomListView` (검색·상태 탭·카테고리 칩) | `livelist` 화면에 해당. 그 앞에 `landing`·`main`이 따로 있다 | **경로 변경** — `/rooms`로 옮긴다 |
| (없음) | `landing` | **신규 구현** |
| (없음) | `main` | **신규 구현** |

| `/onboarding` = `OnboardingView` + 라우터 가드 | `onboard` | 유지 |
| `/rooms/new` = `RoomCreateView` (내부 `step` ref로 1↔2단계) | `create` + `create2` | 유지 — 단계를 `?step=1\|2` 쿼리로 노출만 추가 |
| `/rooms/:id/waiting` | `wait` | 유지 |
| `/rooms/:id/live` | `live` | 유지 |

| `/rooms/:id/result` = `ResultView` (스텁, 낙찰·유찰 겸용 전제) | `result`(역할 4분기) + `nosale`(사유 3분기) 별도 화면 | **구조 재편** — `/rooms/:id/nosale` 신설 |
| `/wallet` = `WalletView` (잔액 카드 2장 + 원장 한 페이지) | 마이페이지 **지갑 탭의 카드 2장** → "거래 내역 보기 →" → **별도 `ledger` 화면** | **구조 재편** (아래 상세) |
| `WalletView` 내 `ChargeModal` / `WithdrawModal` | `charge` · `withdraw` 전체 화면 | **구조 재편** — 라우트로 승격, 직전 화면 복귀 |
| `/me` = `MyPageView` (스텁) | `mypage` 4탭 — 지갑·지금 할 일·경매 목록·리더보드 | **구조 재편** — 지갑 탭이 진입 기본값 |

**지갑 건 (이번 재작성의 발단).** 낡은 표의 `/wallet` = "이중 지갑 + 원장" 한 줄을 근거로
잔액 카드와 원장을 한 페이지에 담은 `WalletView.vue`를 만들었으나, 와이어프레임에 그런 화면은 없다.
실제 구조는 **마이페이지 `지갑` 탭 = 거래/베팅 카드 2장**(잔액 + 충전·출금 버튼)이고,
각 카드 하단의 "거래 내역 보기 →"·"베팅 내역 보기 →"를 누르면 **별도 `ledger` 화면**으로 넘어간다.
`ledger`는 자체 통화 탭(`trade`/`bet`)·잔액 헤더·무한 스크롤을 가지고 "← 마이페이지"로 복귀한다.
따라서 `/wallet` 단독 라우트는 폐지하고, 잔액 카드는 `/me?tab=wallet`으로, 원장은 `/wallet/ledger`로 옮긴다.
`WalletView.vue`가 `docs/04` §3.5에서 끌어낸 판단(available/held 분리 표시, 원장 두 테이블 분리,
홀드 행 라벨링, append-only)은 그대로 유효하므로 **로직은 보존하고 배치만 두 화면으로 나눈다.**

`/wallet/ledger`는 통화별 잔액 헤더만 두고 available/held 상세는 지갑 카드 쪽에 남긴다 —
어느 쪽에 얼마나 둘지는 화면 작업 시 와이어프레임 `ledger`(잔액 1줄)와 대조해 확정한다.

## LiveLayout 프레임 — FE2 인터페이스 계약

`features/live/LiveLayout.vue` (1주차 최우선 산출물, 슬롯 컴포넌트, **FE1 소유**):

  첫 액션 시점에 역할이 확정된다. 상세는 `03-역할-경계-합의.md` §4-1

## 목킹 전략 (`src/mocks/`)

백엔드·미디어 서버가 없는 1~2주차에는 목으로 화면을 돌린다:

- **목 REST** — 방 목록·지갑 잔액 등 정적 JSON. `shared/api/http.ts`의 axios 인스턴스를 거치는 모양만 유지
- **목 라이브 피드** (오너 FE2) — 가짜 STOMP 클라이언트를 만들지 않는다. 만드는 것은 세 가지:
  ① 이벤트 핸들러(`{ type, data }` → 스토어 갱신, **실소켓에서 그대로 재사용되는 진짜 코드**),
  ② 그걸 두드리는 `setInterval` 드라이버(`src/mocks/liveFeed.ts`),
  ③ 스토어 fixture·가짜 action(`placeBid`는 setTimeout으로 `bidStatus` 전이).
  FE1 화면 개발에는 fixture + 가짜 action이면 충분하고, 목 피드는 데모용이다.
  **타이머 틱 이벤트는 없다** — 서버 권위(BID-05) 설계상 카운트다운은 `endsAt` 기준 클라이언트 렌더링이고,
  연장 시 `ENDS_AT_UPDATED`가 한 번 올 뿐이다. 이벤트 타입 목록은 `03-역할-경계-합의.md` §6.

백엔드 WebSocket 공통 포맷은 `{ type: string, data: object }` 래핑이다 (BE2·BE3 명세 합의).
목 이벤트도 처음부터 이 모양으로 만들 것.

## 코딩 규칙

- **색·폰트는 토큰으로만**: `bg-cream`, `text-terra`, `text-on-dark-dim` 등. 하드코딩 hex 금지 (`src/app/styles/main.css`가 토큰 정의)

- 타이머는 항상 서버 권위 — 클라이언트에서 남은 시간을 자체 계산하지 말고 `live` 스토어의 `endsAt` + `serverOffset` 기준으로 표시만 한다 (BID-05)
