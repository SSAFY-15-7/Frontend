# 협업 가이드 (FE1 · FE2)

> 작업 영역, 스토어 계약, 라우트 맵, 목킹 전략. 처음 합류하면 이 문서부터.

## 역할 분담

| 담당 | 영역 | 파일 범위 |
|---|---|---|
| **FE1** | 디자인 시스템·지갑·방 관리·예측 베팅·결과 리포트 | `features/auth·wallet·room·prediction·result/`, `stores/`(live 제외), `shared/` |
| **FE2** | WebRTC 미디어·소켓(STOMP) 인터랙션 | `features/live/` 내부, `stores/live.ts` |

feature 단위로 폴더를 나눈 이유가 파일 충돌 방지다. **자기 영역 밖 파일을 고칠 땐 사전 공유.**

## 라우트 ↔ 화면 매핑

| 경로 | 화면 | 담당 |
|---|---|---|
| `/` | 홈 · 방 목록 (ROOM-02) | FE1 |
| `/login` | 로그인·회원가입·온보딩 (MEM) | FE1 |
| `/rooms/new` | 방 개설 2단계 폼 (ROOM-01, AI 자동완성 연동 예정) | FE1 |
| `/rooms/:id/waiting` | 대기방 — 채팅만 활성, 입찰·베팅 잠금 (ROOM-07) | FE1 프레임 |
| `/rooms/:id/live` | 라이브 뷰 (LIVE·BID·CHAT·PRED) | 프레임 FE1 / 내용 FE2 |
| `/rooms/:id/result` | 낙찰/유찰 결과 — 역할별 4뷰 분기 (DEAL-06·08) | FE1 |
| `/wallet` | 이중 지갑(거래/베팅) + 원장 (PNT) | FE1 |
| `/me` | 마이페이지 — 지금 할 일 + 거래 탭 (DEAL-07) | FE1 |

화면 배치 기준은 기획 레포 `비드라이브 인터랙티브.html`의 해당 화면.

## LiveLayout 프레임 — FE2 인터페이스 계약

`features/live/LiveLayout.vue` (1주차 최우선 산출물, 슬롯 컴포넌트):

- `#video` — WebRTC 스트림 영역 → **FE2가 채움**
- `#chat` — 채팅 패널 영역 → **FE2가 채움**
- `#action` — 입찰/베팅 패널 (입찰↔예측 토글 포함) → FE1
- 헤더(제품 타이틀·타이머 칩)·반응형 배치·영역 크기 배분 → 프레임(FE1) 책임

FE2는 슬롯에 컴포넌트만 꽂으면 되고, 레이아웃·반응형은 신경 쓰지 않아도 된다.

## Pinia 스토어 경계

| 스토어 | 오너 | 내용 |
|---|---|---|
| `auth` | FE1 | 세션, 프로필 |
| `wallet` | FE1 | 거래/베팅 이중 지갑, 원장 |
| `room` | FE1 | 방 목록·개설·대기방 |
| `prediction` | FE1 | 예측 금액·스테이크 입력, 배당 결과 |
| `live` | **FE2** | 소켓 이벤트로 갱신 — 타이머(서버 권위), 입찰 상태, Vibe 피드, 채팅 |

`live` 스토어의 state 모양은 **FE1·FE2 공동 합의 사항**이다. FE1 화면(예측 패널·결과)도 이걸 구독하므로,
구조를 바꿀 땐 반드시 상호 공유 후 진행. 현재 `stores/live.ts`에 합의 초안이 타입으로 잡혀 있다.

> ⚠️ 예측 베팅은 기획서 v1.0에서 **구간 선택 → 금액 직접 입력 + 스테이크(오차율 고정 배당 ×5/×3/×1)**로
> 전면 개편됐다. "구간(bucket)·실시간 분포 그래프"가 나오는 옛 문서·명세를 보면 그쪽이 폐기된 것이다.
> 분포는 라이브 중 노출 금지, 종료 후 결과 화면에서 오차 밴드별로만 공개.

## 목킹 전략 (`src/mocks/`)

백엔드·미디어 서버가 없는 1~2주차에는 목으로 화면을 돌린다:

- **목 REST** — 방 목록·지갑 잔액 등 정적 JSON. `shared/api/http.ts`의 axios 인스턴스를 거치는 모양만 유지
- **목 소켓** — `setInterval` 기반 가짜 이벤트 발행기 (타이머 틱·입찰 이벤트·Vibe·채팅).
  실소켓(STOMP)과 **동일한 이벤트 인터페이스**(`{ type, data }`)로 만들어서, FE2가 실연결로 교체할 때
  스토어·화면 코드는 그대로 두고 발행기만 갈아끼운다

백엔드 WebSocket 공통 포맷은 `{ type: string, data: object }` 래핑이다 (BE2·BE3 명세 합의).
목 이벤트도 처음부터 이 모양으로 만들 것.

## 코딩 규칙

- **색·폰트는 토큰으로만**: `bg-cream`, `text-terra`, `text-on-dark-dim` 등. 하드코딩 hex 금지 (`src/app/styles/main.css`가 토큰 정의)
- 공통 컴포넌트는 미리 만들지 않는다 — 같은 UI가 **두 번째로 등장할 때** `shared/ui/`로 추출
- 금액·타이머 등 숫자는 tabular numbers로 (자릿수 흔들림 방지)
- 타이머는 항상 서버 권위 — 클라이언트에서 남은 시간을 자체 계산하지 말고 `live` 스토어의 `endsAt` + `serverOffset` 기준으로 표시만 한다 (BID-05)
