# 협업 가이드 (FE1 · FE2)

> 작업 영역, 스토어 계약, 라우트 맵, 목킹 전략. 처음 합류하면 이 문서부터.
> 인터페이스 계약(스토어 시그니처·슬롯·props·이벤트 타입)의 단일 소스는 `03-역할-경계-합의.md`.

## 역할 분담

| 담당 | 영역 | 파일 범위 |
|---|---|---|
| **FE1** | 디자인 시스템·지갑·방 관리·입찰/예측 패널·결과 리포트·라이브 프레임 | `features/auth·wallet·room·bid·prediction·result/`, `features/live/LiveView·LiveLayout.vue`, `stores/`(live 제외), `shared/` |
| **FE2** | WebRTC 미디어·소켓(STOMP)·채팅·대기방 | `features/live/media·socket/`, `features/chat/`, `stores/live.ts` |

feature 단위로 폴더를 나눈 이유가 파일 충돌 방지다. **자기 영역 밖 파일을 고칠 땐 사전 공유.**
상대 소유 컴포넌트는 import는 자유, 내부 수정만 합의 대상이다.

## LiveLayout 프레임 — FE2 인터페이스 계약

`features/live/LiveLayout.vue` (1주차 최우선 산출물, 슬롯 컴포넌트, **FE1 소유**):

- `#video` — WebRTC 스트림 영역 → **FE2가 채움** (`features/live/media/`)
- `#chat` — 채팅 패널 영역 → **FE2가 채움** (`features/chat/ChatPanel.vue`, 대기방과 공용)
- `#action` — 입찰/예측 패널 → FE1 (`features/bid/`·`features/prediction/`).
  **탭 전환이 아니다** — 한 방에서 입찰과 예측 중 하나만 참여할 수 있고,
  첫 액션 시점에 역할이 확정된다. 상세는 `03-역할-경계-합의.md` §4-1
- 헤더(제품 타이틀·타이머 칩)·반응형 배치·영역 크기 배분 → 프레임(FE1) 책임

FE2는 슬롯에 컴포넌트만 꽂으면 되고, 레이아웃·반응형은 신경 쓰지 않아도 된다.
슬롯 안 컴포넌트는 부모를 100% 채우게만 만들고 자체 크기·위치를 잡지 않는다.
ChatPanel props 계약은 `03-역할-경계-합의.md` §5.

## Pinia 스토어 경계

| 스토어 | 오너 | 내용 |
|---|---|---|
| `auth` | FE1 | 세션, 프로필 |
| `wallet` | FE1 | 거래/베팅 이중 지갑, 원장 |
| `room` | FE1 | 방 목록·개설·대기방 |
| `prediction` | FE1 | 예측 금액·스테이크 입력, 배당 결과 |
| `live` | **FE2** | 소켓 이벤트로 갱신 — 타이머(서버 권위), 입찰 상태, Vibe 피드, 채팅 |

`live` 스토어의 state·action 시그니처는 **FE1·FE2 공동 합의 사항**이다. FE1 화면(입찰·예측 패널·결과)도
이걸 구독·호출하므로, 구조를 바꿀 땐 반드시 상호 공유 후 진행.
합의된 시그니처(`placeBid`·`sendChat`·`connect`·`bidStatus` 등)는 `03-역할-경계-합의.md` §3이 단일 소스다 —
구현은 목(가짜 action)에서 실소켓으로 바뀌어도 시그니처는 불변.

> ⚠️ 예측 베팅은 기획서 v1.0에서 **구간 선택 → 금액 직접 입력 + 스테이크(오차율 고정 배당 ×5/×3/×1)**로
> 전면 개편됐다. "구간(bucket)·실시간 분포 그래프"가 나오는 옛 문서·명세를 보면 그쪽이 폐기된 것이다.
> 분포는 라이브 중 노출 금지, 종료 후 결과 화면에서 오차 밴드별로만 공개.

## 목킹 전략 (`src/mocks/`)

백엔드·미디어 서버가 없는 1~2주차에는 목으로 화면을 돌린다:

- **목 REST** — 방 목록·지갑 잔액 등 정적 JSON. `shared/api/http.ts`의 axios 인스턴스를 거치는 모양만 유지
- **목 라이브 피드** (오너 FE2) — 가짜 STOMP 클라이언트를 만들지 않는다. 만드는 것은 세 가지:
  ① 이벤트 핸들러(`{ type, data }` → 스토어 갱신, **실소켓에서 그대로 재사용되는 진짜 코드**),
  ② 그걸 두드리는 `setInterval` 드라이버(`src/mocks/liveFeed.ts`),
  ③ 스토어 fixture·가짜 action(`placeBid`는 setTimeout으로 `bidStatus` 전이).
  FE1 화면 개발에는 fixture + 가짜 action이면 충분하고, 목 피드는 데모용이다.
  **타이머 틱 이벤트는 없다** — 서버 권위(BID-05) 설계상 카운트다운은 `endsAt` 기준 클라이언트 렌더링이고,
  연장 시 `ENDS_AT_UPDATED`가 한 번 올 뿐이다. 이벤트 타입 목록은 `03-역할-경계-합의.md` §6.

백엔드 WebSocket 공통 포맷은 `{ type: string, data: object }` 래핑이다 (BE2·BE3 명세 합의).
목 이벤트도 처음부터 이 모양으로 만들 것.

## 코딩 규칙

- **색·폰트는 토큰으로만**: `bg-cream`, `text-terra`, `text-on-dark-dim` 등. 하드코딩 hex 금지 (`src/app/styles/main.css`가 토큰 정의)
- 공통 컴포넌트는 미리 만들지 않는다 — 같은 UI가 **두 번째로 등장할 때** `shared/ui/`로 추출
- 금액·타이머 등 숫자는 tabular numbers로 (자릿수 흔들림 방지)
- 타이머는 항상 서버 권위 — 클라이언트에서 남은 시간을 자체 계산하지 말고 `live` 스토어의 `endsAt` + `serverOffset` 기준으로 표시만 한다 (BID-05)
