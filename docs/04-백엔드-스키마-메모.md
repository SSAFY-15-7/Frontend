# 백엔드 스키마 FE 참조 메모

> 원본: `C:\Users\SSAFY\gongtong\bidlive_postgresql_schema.sql` (PostgreSQL 14+, 22개 테이블) — 2026-07-22 확인 기준.
> **이 문서는 파생물이고, 충돌하면 항상 원본 SQL이 우선한다.** 스키마가 갱신되면 이 문서도 같이 갱신해야 한다.
> 목적은 두 가지다. ① 화면을 만들 때 스키마를 매번 다시 읽지 않게 하는 것, ② DB 제약이 UI 요구사항으로
> 번역된 것을 놓치지 않게 하는 것. 테이블 DDL 전문이 필요하면 원본을 보라 — 여기 옮겨 적지 않는다.
> 이 문서는 **기록만 한다.** 코드 수정은 각 화면 작업에서 별도로 한다 (§4 참조).

## 1. 화면별 매핑

"주 테이블"은 화면의 뼈대가 되는 것, "참조"는 조인해서 같이 필요한 것.

> ⚠️ 아래 표는 **구 라우트 표(9행) 기준**이다. `docs/02-협업-가이드.md`의 라우트 표가
> 와이어프레임 17개 화면 기준으로 재작성되면서 `landing`·`main`·`console`·`nosale`·`ledger`·`charge`·`withdraw`가
> 추가됐고 `/wallet`이 폐지됐다. 이 표는 아직 갱신되지 않았다 — 화면별 테이블 매핑은
> 각 화면 작업 시 새 라우트 표와 대조해 채운다.

- **방 목록**: `auction_room`에는 제목이 없다. 카드에 뿌릴 제목·카테고리·썸네일은 전부
  `product` → `product_category` / `product_image`(`is_thumbnail = TRUE`) 조인 결과다. 입찰 건수·참여자 수는
  어느 테이블에도 컬럼으로 없고 `bid`/`auction_participant` 집계이므로, 목록 API가 집계를 내려주는지 확인이 필요하다(§6).
- **라이브 뷰**: `#action` 슬롯이 읽는 값은 `wallet`(잔액), `auction_participant`(내 참여 유형),

- **결과 화면**: `auction_result` 1행 + 내 `prediction_result` + (낙찰이면) `deal`·`deal_shipping`.

- **마이페이지**: DEAL-07 "지금 할 일"은 `deal.deal_status` + `deal_shipping`의 NULL 여부로 계산된다
  (주소 미입력 / 발송 대기 / 수취 확인 대기). `deal_status` 값 목록은 미확인이다(§6).

`report`는 어느 라우트에도 대응 화면이 없다. `point_charge`는 `/wallet`에 붙을 후보지만 결제 리다이렉트 경로가

## 3. UI 제약으로 번역된 DB 제약

**이 문서에서 가장 중요한 섹션이다.** 각 항목은 "제약 → 그래서 화면은 이래야 한다" 형태다.
"추정"으로 표시된 것은 스키마만으로는 단정할 수 없어 §6에서 확인이 필요한 항목이다.

### 3.1 참여·입찰·예측 (BID·PRED)

**`auction_participant`에 `UNIQUE(auction_room_id, member_id)`가 걸려 있다.** 한 회원은 한 방에서 참가 행을
정확히 하나만 가진다. 그리고 `participation_type`은 `BIDDER` 또는 `PREDICTOR` 둘 중 하나다.
즉 **한 방에서 입찰과 예측을 동시에 할 수 없고, 한 번 정한 역할을 바꾸려면 기존 행을 UPDATE해야 한다.**
이것은 `docs/02`·`docs/03`이 전제한 "#action 슬롯의 입찰↔예측 토글"과 정면으로 부딪힌다.
토글이 단순 화면 전환이 아니라 역할 변경 요청이 되기 때문이다. 세 가지 선택지가 있다.

1. 입장 시 역할을 고르게 하고(모달) 이후 `#action`은 고른 역할의 패널만 보여준다. 토글 제거.
2. 토글은 유지하되 반대편 패널은 읽기 전용으로 잠그고, 전환 시 "역할을 바꾸시겠습니까" 확인을 받는다.
   이미 `bid`나 `prediction`을 제출했다면 전환 불가로 막아야 한다(FK RESTRICT 때문에 참가 행 삭제가 불가능하다).
3. 백엔드가 역할 전환을 허용하지 않는다면 토글 자체를 없앤다.

**결정은 사람이 한다.** 다만 이 제약을 모르고 토글을 구현하면 3주차 REST 연동 때 되돌려야 한다.

**`prediction.auction_participant_id`가 UNIQUE다.** 방당 예측 베팅은 1회뿐이다. 이미 베팅했다면 금액·스테이크
입력 폼을 잠그고 "확정됨: 예측가 X / 스테이크 Y" 표시로 전환해야 한다. 수정·취소 버튼을 두면 안 된다.

**반면 `bid`는 `UNIQUE(auction_participant_id, sequence_no)`다.** 시퀀스 번호가 다르면 같은 참가자가 여러 번
입찰할 수 있다. 입찰 패널은 반복 제출 가능한 폼이고, 내 입찰 이력이 여러 건 쌓인다.
`sequence_no > 0`이므로 1부터 시작한다. 누가 시퀀스를 매기는지는 미확인이다(§6).

**`bid.idempotency_key`와 `prediction.idempotency_key`는 둘 다 `NOT NULL UNIQUE`다.**
같은 키로 두 번 요청하면 두 번째는 DB 레벨에서 거부된다. 이는 클라이언트가 키를 생성해 보내는 설계일
가능성이 높다(추정) — 그래야 네트워크 재시도가 중복 입찰이 되지 않는다. **생성 주체는 미확정이다(§6).**
클라이언트 생성이라면 제출 버튼을 누르는 시점에 UUID를 만들고, 재시도 시 같은 키를 재사용해야 한다.
`trade_point_ledger`·`betting_point_ledger`에도 같은 제약이 있다.

**`bid.amount > 0`, `prediction.predicted_price > 0`, `prediction.stake_amount > 0`.**
0원 입력은 전부 거부된다. 스테이크 0으로 "무료 참여"를 만들 수 없다.

**스키마에는 `bid.amount >= auction_room.reserve_price` 제약이 없다.** 시작가 미만 입찰의 유효성은 서버
애플리케이션 로직이 판단한다(추정). FE는 힌트(플레이스홀더·헬퍼 텍스트)로 안내하되, 최종 판정은
`bidStatus`의 `rejected` + `bidRejectReason`으로 받는다. 최소 입찰 단위(호가 단위)도 스키마에 없다(§6).

### 3.2 예측 결과 (PRED)

**`prediction_result.reward_multiplier IN (0, 1, 3, 5)`.** 결과 UI는 **4가지 경우**를 다뤄야 한다.
AGENTS.md 규칙 7의 ×5/×3/×1에 **미적중(×0)이 추가된 것**이다.
현재 `src/stores/prediction.ts`의 `BAND_MULTIPLIER`(`WITHIN_1: 5`, `WITHIN_3: 3`, `WITHIN_5: 1`, `MISS: 0`)와
값 집합은 일치한다. 다만 **밴드 경계(1%/3%/5%)는 스키마 어디에도 없다** — 서버가 `error_rate`로 판정한
`reward_multiplier`를 그대로 받아 쓰는 것이 안전하고, FE가 `error_rate`로 밴드를 재계산하면 경계 처리
(이상/초과)에서 서버와 어긋날 수 있다.

**`error_rate NUMERIC(8,6)`, `CHECK (error_rate >= 0)`.** 음수가 없다 = 절대 오차율이다.
예측가가 낙찰가보다 높았는지 낮았는지는 이 값으로 알 수 없으므로, "얼마나 높게/낮게 예측했는지"를 보여주려면
`prediction.predicted_price`와 `auction_result.final_price`를 직접 비교해야 한다.
소수 6자리이므로 값이 비율(0.05 = 5%)인지 퍼센트(5.0 = 5%)인지 확인이 필요하다(§6).

**`payout_amount >= 0`.** 미적중이면 0이 저장된다. NULL이 아니므로 "정산 전"과 "미적중 0원"을 이 값으로
구분할 수 없다 — 정산 여부는 `prediction_result` 행의 존재 유무로 판단한다.

**`prediction_result.prediction_id`가 UNIQUE.** 예측 1건에 결과 1건. 재정산으로 행이 늘지 않는다.

### 3.3 경매 결과 (BID·DEAL)

**`auction_result.result_status IN ('SOLD', 'NO_SALE', 'CANCELLED')`.**
결과 화면은 낙찰/유찰 2분기가 아니라 **3분기**다. `CANCELLED`(취소)를 별도로 다뤄야 하고,
`docs/02`가 말하는 "역할별 4뷰"와 곱해지면 조합이 더 늘어난다.

**`ck_auction_result_status_fields`**: `SOLD`면 `winner_participant_id`·`winning_bid_id`·`highest_bid_amount`·
`final_price`가 모두 NOT NULL, `NO_SALE`/`CANCELLED`면 `winner_participant_id`·`winning_bid_id`·`final_price`가
모두 NULL이다. 결과 화면에서 낙찰가·낙찰자 영역은 `SOLD`에서만 렌더해야 하고, 그 외엔 옵셔널 체이닝이 아니라
아예 분기해야 한다. **주의: `highest_bid_amount`는 이 제약에서 `NO_SALE` 쪽에 언급되지 않는다** — 유찰이어도
최고 입찰가가 존재할 수 있다(유찰 사유가 "시작가 미달"인 경우 등, 추정).

**`highest_bid_amount`·`second_highest_bid_amount`·`final_price`가 별도 컬럼이다.**
비크리(2등 가격) 구조상 `final_price`가 `highest_bid_amount`와 다를 수 있다. 결과 화면은
"최고가"와 "실제 낙찰가"를 **구분해서** 표기해야 하고, 하나로 합치면 비크리의 핵심이 사라진다.
`second_highest_bid_amount`를 화면에 노출할지는 정책 판단이 필요하다(§6) — 종료 후이므로 규칙 6 위반은 아니다.

**`valid_bidder_count`·`total_bid_count`, `total_bid_count >= valid_bidder_count`.**
"입찰 N건 / 유효 입찰자 M명"으로 두 값을 함께 보여줄 수 있다. 라이브 중 공개 가능한 값(건수·참여자 수)과
같은 성격이므로 규칙 6에 걸리지 않는다.

**`auction_result.auction_room_id`가 UNIQUE, `winner_participant_id`·`winning_bid_id`는 부분 UNIQUE 인덱스.**
방당 결과 1건, 낙찰자·낙찰 입찰은 전 시스템에서 한 번만 쓰인다. 결과 화면은 `roomId`로 단건 조회하면 된다.

### 3.4 방·상품 (ROOM·LIVE)

**`auction_room.start_type IN ('IMMEDIATE','SCHEDULED')` + `ck_auction_room_schedule`**
(IMMEDIATE면 `scheduled_at` NULL, SCHEDULED면 NOT NULL).
방 개설 폼(ROOM-01)에 **즉시 시작 / 예약 시작 선택 UI가 필요**하고, 예약을 고르면 시각 입력이 필수 검증이다.
즉시를 골랐다가 예약으로 바꾸면 시각 필드를 다시 채워야 하고, 반대로 바꿀 땐 시각을 비워 보내야 한다.

**`auction_room.extension_count BETWEEN 0 AND 3`.** BID-03 연장은 최대 3회.
판매자 화면의 연장 버튼은 3회 소진 시 비활성화하고 "연장 3/3 소진"을 표시해야 한다.
`live` 스토어의 `extensionCount`가 이 값이고, `ENDS_AT_UPDATED` 이벤트로 갱신된다(`docs/03` §6).

**`auction_room.reserve_price > 0`, `duration_minutes > 0`.** 시작가·진행 시간 모두 0 불가. 최솟값은 1이다.

**`auction_room.product_id`가 UNIQUE.** 상품 하나당 방 하나다. "재경매" 기능을 만든다면 기존 상품을 재사용할 수
없고 상품을 새로 등록하는 흐름이 된다.

**`started_at`·`ends_at`·`ended_at`이 전부 nullable.** 시작 전(대기방)에는 `ends_at`이 NULL이다.
**대기방에서 종료 카운트다운을 그릴 수 없다** — 대기방에 표시할 수 있는 건 `scheduled_at`(시작 예정) 뿐이다.
타이머 컴포넌트를 재사용할 때 `endsAt`이 NULL인 상태를 반드시 다뤄야 한다.

**`openvidu_session_id`에 부분 UNIQUE 인덱스(NOT NULL인 행만).** 방당 세션 1개, 세션 ID는 전역 유일.
NULL 가능이므로 방송 시작 전에는 세션이 없다 — FE2의 미디어 연결은 이 값이 도착한 뒤에만 시도해야 한다.

**`product.moderation_status IN ('PENDING','PASSED','FLAGGED','BLOCKED')`, `product_image`도 동일.**
등록 직후는 `PENDING`이다. 방 개설 완료 화면에 "검수 중" 상태 표시가 필요하고,
`BLOCKED` 상품·이미지가 목록/상세에 어떻게 나타나는지(제외인지 가림 처리인지) 확인이 필요하다(§6).
이미지는 **장별로** 검수 상태를 가지므로, 일부 이미지만 가려지는 경우를 갤러리 UI가 처리해야 한다.

**`product.description`이 `NOT NULL TEXT`.** 방 개설 폼에서 설명은 필수 입력이다. 빈 문자열 방지 검증이 필요하다
(`title`은 `VARCHAR(100)`으로 길이 제한, `product_category.name`은 50자).

**`product.condition_status`는 nullable이고 `IN ('NEW','LIKE_NEW','USED','DAMAGED')`.**
상품 상태 선택 UI는 4지선다 + "선택 안 함"이 유효하다. 필수로 만들면 스키마보다 엄격해진다.

**`product_image`: `UNIQUE(product_id, sort_order)` + `sort_order > 0`.**
이미지 순서는 1부터, 중복 없이 보내야 한다. 드래그 재정렬 UI를 만든다면 전체 순서를 재계산해서 보내는 편이 안전하다.

**`uq_product_single_thumbnail` (부분 UNIQUE 인덱스, `WHERE is_thumbnail = TRUE`).**
대표 이미지는 **최대 1장**이다. 체크박스 여러 개가 아니라 **라디오 버튼**이어야 한다.
0장도 허용되므로 목록 카드는 썸네일 없는 경우(`sort_order` 최솟값으로 폴백 등)를 다뤄야 한다.

**`product_category.code`가 UNIQUE.** 카테고리는 서버가 관리하는 마스터 데이터다.
현재 목 데이터가 쓰는 `'전자기기'` 같은 한글 문자열은 `name`이고, 필터·전송에는 `code`를 써야 한다(추정).

### 3.5 포인트·지갑 (PNT)

**`wallet`: `UNIQUE(member_id, wallet_type)`, `wallet_type IN ('TRADE','BETTING')`,
그리고 `available_balance`와 `held_balance`가 분리되어 있다.**
지갑 화면은 지갑마다 **"사용 가능 / 보류" 두 값을 따로** 표시해야 한다. 단일 잔액으로 그리면 안 된다.
`available_balance`는 **이미 보류분을 제외한 값**이다 — 여기서 다시 보류를 빼면 이중 차감이다(§4 참조).
총 보유는 `available_balance + held_balance`다.
두 지갑은 별개 행이며 상호 전환 개념이 스키마에 없다. 화면에서도 전환 UI를 만들지 않는다.

**`available_balance >= 0`, `held_balance >= 0`.** 마이너스 잔액이 없다.
입찰·베팅 폼은 제출 전에 잔액 초과를 막아야 하고, 초과 요청은 서버에서 거부된다.

**원장이 `trade_point_ledger`와 `betting_point_ledger` 두 테이블로 나뉘어 있다.**
지갑 화면의 원장 탭도 거래/베팅으로 분리하는 것이 자연스럽다. 하나의 통합 타임라인을 만들려면
두 테이블을 FE에서 병합·정렬해야 하는데, 페이지네이션이 깨진다.

**두 원장 모두 `available_delta`와 `held_delta`를 함께 가지고, `CHECK (available_delta <> 0 OR held_delta <> 0)`.**
입찰 홀드는 `available_delta = -X`, `held_delta = +X`처럼 **총합이 변하지 않는 행**으로 기록된다.
원장 행을 단일 금액으로 렌더하면 이런 행이 "증감 0"으로 보인다. **두 델타를 모두 보여주거나,
`transaction_type`에 따라 "홀드", "홀드 해제" 같은 라벨로 표현해야 한다.**
`available_balance_after`·`held_balance_after`가 함께 저장되므로 각 행 시점의 잔액을 표시할 수 있다.

**원장의 FK로 상세 이동이 가능하다.** `trade_point_ledger`는 `auction_participant`·`bid`·`deal`·`point_charge`를,
`betting_point_ledger`는 `prediction`·`prediction_result`를 참조한다.
원장 행에서 해당 방·거래·충전 내역으로 딥링크하는 UI가 가능하다(전부 nullable이므로 없는 경우 처리 필요).

**`point_charge`: `order_id` UNIQUE, `payment_key` UNIQUE(nullable).**
결제 실패 후 재시도는 새 `order_id`로 시작한다 — 같은 주문번호 재사용 UI를 만들면 안 된다.
**`charge_point_amount`와 `payment_amount`가 별도 컬럼**이므로, 충전 포인트 수량과 실제 결제 금액이 다를 수 있다
(프로모션·보너스, 추정). 충전 화면은 두 값을 각각 표시해야 한다.
`failure_code`·`failure_message`가 있으므로 실패 사유 표시 UI가 필요하고,
`approved_at`/`cancelled_at`이 nullable이라 "요청됨 → 승인/취소" 중간 상태가 존재한다.

### 3.6 회원·약관 (MEM)

**`member.nickname`은 UNIQUE이면서 nullable, `VARCHAR(30)`.**
닉네임은 전역 중복 불가 → 온보딩 폼에 중복 확인과 409 처리가 필요하고, 입력 길이는 30자로 제한한다.
nullable이라는 것은 **닉네임이 없는 회원 상태가 정상적으로 존재한다**는 뜻이다 —
카카오 로그인 직후 닉네임 미설정 = 온보딩 미완료. 라우터 가드로 온보딩 화면에 붙잡아야 한다(추정).

**`member.status IN ('ACTIVE','SUSPENDED','WITHDRAWN')` + `ck_member_withdrawn_at`**
(WITHDRAWN이면 `withdrawn_at` NOT NULL, 아니면 NULL).
`SUSPENDED` 회원이 접속했을 때의 화면이 필요하다 — 입찰·베팅·채팅을 잠그고 정지 안내를 띄운다.
탈퇴는 소프트 삭제이므로 탈퇴 회원의 과거 데이터가 남는다.

**`member.role IN ('USER','ADMIN')`.** 관리자 화면이 스키마상 전제되어 있다(`report.processed_by_member_id`,
`chat_message.moderation_*`). MVP 범위인지는 확인 필요(§6).

**`social_account.provider`에 `CHECK (provider = 'KAKAO')`.**
로그인 화면에 **카카오 버튼 하나만** 둔다. 구글·애플 버튼을 만들면 저장할 수 없다.
`UNIQUE(member_id, provider)`이므로 계정 추가 연동 UI도 불필요하다.

**`terms`: `UNIQUE(terms_type, version)`, `is_required`, `is_active`, `effective_at`.**
약관은 버전 관리된다. 동의 화면은 **필수/선택을 구분**해서, 필수 약관 미동의 시 다음 단계를 막아야 한다.
`is_active` + `effective_at` 인덱스가 있으므로 "현재 유효한 약관 목록"을 서버가 내려준다(추정).

**`member_terms_agreement`: `UNIQUE(member_id, terms_id)` + `ck_member_terms_agreement_state`.**
같은 약관 버전에 대한 동의는 회원당 1행이다(재동의는 UPDATE).
상태 제약이 정의하는 유효 조합은 세 가지다 — 동의(`is_agreed=TRUE`, `agreed_at` 있음, `withdrawn_at` 없음),
미동의(`is_agreed=FALSE`, `agreed_at` 없음), 철회(`is_agreed=FALSE`, `withdrawn_at` 있음).
**"동의 철회"가 스키마에 명시적으로 존재하므로 마이페이지에 철회 UI가 필요하다**(선택 약관 한정, 추정).
새 버전이 나오면 `terms_id`가 달라 새 행이 생기므로, 재동의 요구 화면이 필요하다.

### 3.7 채팅 (CHAT)

**`chat_message.sender_member_id`가 nullable이고 `ON DELETE SET NULL`.**
탈퇴 회원의 메시지는 발신자가 NULL이 된다. 채팅 렌더러는 **닉네임 없는 메시지**를 반드시 처리해야 한다
("알 수 없음" 또는 시스템 메시지). NULL을 그대로 표시하면 화면이 깨진다.

**`is_pinned`, `question_queue_status`, `is_deleted`, `moderation_status`, `moderation_reason`.**
채팅 패널이 다뤄야 하는 상태가 단순 텍스트 나열보다 많다.

- `is_pinned` → 공지 고정 영역이 필요하다.
- `question_queue_status` → CHAT-06 질문 큐. nullable이므로 일반 메시지는 NULL, 질문만 값을 가진다.
  `docs/03` §5의 `highlightQuestions` prop이 이것과 대응된다. 값 목록은 미확인(§6).
- `is_deleted` → 삭제된 메시지 자리 표시("삭제된 메시지입니다") 또는 필터링.
- `moderation_status`/`moderation_reason` → 가려진 메시지에 사유를 함께 보여줄 수 있다.

**`CHECK (length(btrim(content)) > 0)`.** 공백만 있는 메시지는 저장되지 않는다.
전송 버튼은 trim 후 빈 문자열이면 비활성화해야 한다.

### 3.8 거래·배송 (DEAL)

**`deal.auction_result_id`가 UNIQUE.** 결과당 거래 1건.

**`CHECK (seller_member_id <> buyer_member_id)`.** 판매자가 자기 상품을 낙찰받을 수 없다.
방 진입 시 판매자 본인에게는 입찰 UI를 숨기거나 잠가야 한다(예측 참여 가능 여부는 확인 필요, §6).

**`CHECK (NOT (completed_at IS NOT NULL AND cancelled_at IS NOT NULL))`.**
완료와 취소는 배타적이다. 거래 상태 배지는 둘 중 하나만 나온다.

**`deal_shipping`의 주소 필드가 전부 nullable이고 `deal_id`가 UNIQUE.**
거래 생성 직후에는 배송지가 비어 있다 → **"배송지 입력" CTA가 있는 단계가 존재**한다.
`address_submitted_at` → `shipped_at` → `received_confirmed_at` 순서 제약이 있으므로,
이 세 타임스탬프의 NULL 여부로 거래 진행 단계 UI(스테퍼)를 그릴 수 있다.

**`auto_confirm_at`.** 자동 구매확정 시각이 서버 값으로 내려온다. 남은 시간 표시는 이 값 기준으로 하고,
클라이언트에서 자체 계산하지 않는다 — AGENTS.md 규칙 5와 같은 원칙이다.

**`carrier`·`tracking_number`.** 판매자용 송장 입력 폼과 구매자용 배송 조회 표시가 필요하다.

### 3.9 리더보드 (PRED)

**`prediction_leaderboard`: `UNIQUE(ranking_date, member_id)` + `UNIQUE(ranking_date, rank_no)`.**
일자별 스냅샷이며, 같은 날짜에 같은 순위 번호가 두 번 나올 수 없다.
**공동 순위(1위 두 명) 표시가 불가능하다** — 동점자도 서버가 강제로 순서를 매긴다.
정렬 인덱스가 `(score DESC, average_error_rate ASC, member_id)`이므로 타이브레이커는 평균 오차율이다(추정).

**`rewarded_count + lost_count <= prediction_count`.** 합계가 같지 않아도 되므로
"정산 대기 중인 예측"이 존재할 수 있다. 적중률을 `rewarded / prediction_count`로 계산하면 미정산 건 때문에
낮게 나온다 — `rewarded / (rewarded + lost)`가 더 정확하다(추정).

**`best_error_rate <= average_error_rate`.** 두 값을 같이 보여줄 때 순서가 뒤집힐 일은 없다.

### 3.10 신고 (운영)

**`report`: `ck_report_target_count`(대상 4개 중 정확히 1개만 NOT NULL) +
`ck_report_target_type`(`target_type`과 채워진 대상이 일치해야 함).**
신고 대상은 `MEMBER` / `AUCTION_ROOM` / `CHAT_MESSAGE` / `DEAL` 네 종류다.
**신고 진입점이 화면상 네 군데 필요하다** — 프로필, 방 상세, 채팅 메시지 컨텍스트 메뉴, 거래 상세.
신고 폼은 대상 종류를 사용자가 고르는 것이 아니라 진입점이 결정한다.

**`reason_code VARCHAR(40)`에 CHECK가 없다.** 사유 목록을 FE가 하드코딩할 수 없다 — 서버에서 받아야 한다(§6).

## 4. 현재 FE 코드와의 불일치

**이 섹션은 기록만 한다. 코드 수정은 각 화면 작업에서 별도로 진행한다.**

### 4.1 `src/mocks/rooms.ts` — `Room` 타입

| `state: 'live'\|'waiting'\|'ended'` | `auction_room.status` (CHECK 없음) | 서버 값 목록 확인 후 매핑 테이블 작성 (§6) |

| `endsAt?: number` (epoch ms) | `auction_room.ends_at TIMESTAMPTZ` | 절대 시각이라는 점은 일치(규칙 5). 서버가 ISO 문자열을 준다면 변환 지점이 필요 (§6.3) |
| `startsAtLabel?: string` | `auction_room.scheduled_at` | 서버는 시각을, FE가 포맷. 라벨 문자열을 API로 받지 않는다 |

| 없음 | `product.moderation_status` | 검수 상태 배지가 필요하면 추가 |
| 없음 | `auction_room.start_type` / `scheduled_at` | 예약 방 구분 표시에 필요 |

`RoomState`가 3값인데 `auction_result.result_status`는 `SOLD`/`NO_SALE`/`CANCELLED` 3값이고,
`auction_room.status`는 값 목록 미상이다. `ended` 하나로 낙찰·유찰·취소를 뭉뚱그리면 카드에서 구분할 수 없다.

### 4.2 `src/stores/wallet.ts`

| 지금 FE | 스키마 | 조치 |
| --- | --- | --- |
| `tradePoint` / `tradeHold` | `wallet(TRADE).available_balance` / `held_balance` | 필드명 정렬 |
| `betPoint` (단일 값) | `wallet(BETTING)`도 `available` + `held` 2값 | **베팅 지갑의 보류 잔액이 누락됨.** 필드 추가 필요 |
| `tradeAvailable = tradePoint - tradeHold` | `available_balance`가 이미 보류 제외 값 | **이중 차감 버그 위험.** 매핑 규칙 확정 필요 |
| 없음 | 원장 (`trade_point_ledger` / `betting_point_ledger`) | 원장 state·액션 미구현 |
| 없음 | `point_charge` | 충전 요청·상태 미구현 |

`tradeAvailable` getter는 `tradePoint`를 "총 보유"로 해석할 때만 맞다. 스키마의 `available_balance`를
`tradePoint`에 그대로 넣으면 보류를 두 번 빼게 된다. 어느 쪽 의미로 통일할지 정해야 한다.

### 4.3 `src/stores/live.ts` (FE2 소유 — 제안만, 임의 수정 금지)

(별개 이슈). `bidCount`/`participantCount`는 스키마상 집계값이므로 소켓 이벤트로만 들어온다.

### 4.4 `src/stores/prediction.ts`

| 지금 FE | 스키마 | 조치 |
| --- | --- | --- |
| `BAND_MULTIPLIER` = 5/3/1/0 | `reward_multiplier IN (0,1,3,5)` | **값 집합 일치.** 유지 |
| `PredictionBand` = `WITHIN_1`/`WITHIN_3`/`WITHIN_5`/`MISS` | 밴드 개념·경계값이 스키마에 없음 | 서버가 준 `reward_multiplier`를 밴드로 역매핑하는 방식이 안전 |
| `myPrediction` / `myStake` | `prediction.predicted_price` / `stake_amount` | 필드명 정렬. 제출 후 잠금 상태 플래그 필요 (§3.1) |
| `bands[].participantCount` / `totalPayoutAmount` | `prediction_result` 집계 | 집계 API 필요 |
| 없음 | `prediction_result.error_rate` | 내 오차율 표시 필드 없음 |
| 없음 | `prediction_result.payout_amount` | 내 실수령액 필드 없음 (현재는 추정치 getter만 있음) |
| 없음 | `idempotency_key` | 제출 멱등키 미구현 (§6) |

`estimatedPayout` getter는 `stake × multiplier`인데, 서버의 `payout_amount`가 원금 포함인지 순수익인지 확인이
필요하다(§6). `WITHIN_5: 1`을 "원금 반환"으로 주석 달아둔 것을 보면 원금 포함으로 가정하고 있다.

### 4.6 `src/stores/room.ts`

스키마 대비 누락이라기보다 미구현 상태다. `loadRooms()` 하나뿐이고 방 개설(ROOM-01)에 필요한

불일치 항목은 총 **37건**이다 (§4.1 15건, §4.2 5건, §4.3 7건, §4.4 6건, §4.5 4건).

## 5. 신규 테이블이 만드는 새 요구사항

`docs/02`의 라우트 맵에 대응 화면이 없는 것들이다. **선택지만 제시한다. 결정은 사람이 한다.**

### 5.1 약관 (`terms` + `member_terms_agreement`) — MEM

필요한 화면 기능은 세 가지다.

1. **가입 시 약관 동의 단계** — 필수/선택 구분, 전문 보기, 필수 미동의 시 진행 차단.
2. **약관 전문 조회** — `terms.content`가 TEXT이므로 모달 또는 별도 페이지.
3. **동의 이력 조회·철회** — `withdrawn_at`이 스키마에 있으므로 철회 경로가 전제된다.
4. **약관 개정 시 재동의** — 새 `version` 행이 생기면 기존 동의로 커버되지 않는다.

붙일 후보:

| 후보 | 장점 | 단점 |
| --- | --- | --- |
| `/login` 온보딩 안에 단계 추가 | 새 라우트 없음, 가입 흐름과 자연스러움 | 로그인 화면이 무거워짐. 재동의 케이스를 담기 애매함 |
| `/onboarding` 신설 (로그인 후 리다이렉트) | 닉네임 설정 + 약관 동의를 한 곳에 (닉네임도 nullable이라 온보딩이 필요함) | 라우트 1개 추가, 가드 로직 필요 |
| `/terms/:type` 전문 조회 페이지 | 딥링크·푸터 링크 가능 | 모달로도 충분할 수 있음 |
| `/me` 안에 "약관 동의 현황" 섹션 | 라우트 추가 없음 | 마이페이지가 이미 "지금 할 일 + 거래 탭"으로 붐빔 |

`member.nickname`이 nullable인 것과 묶으면 **온보딩 라우트 신설이 가장 정합적이다(제안, 결정 아님)** —
닉네임 설정과 약관 동의가 같은 게이트에 걸린다.

### 5.2 예측 리더보드 (`prediction_leaderboard`) — PRED

일자별 스냅샷 테이블이고 인덱스가 `(ranking_date DESC, score DESC, ...)`와 `(member_id, ranking_date DESC)`
두 방향으로 걸려 있다. 즉 **전체 랭킹 조회**와 **내 이력 조회** 두 가지 화면을 서버가 상정하고 있다(추정).

| 후보 | 장점 | 단점 |
| --- | --- | --- |
| `/leaderboard` 신설 | 전체 랭킹을 제대로 보여줄 공간 확보. 날짜 선택 가능 | 라우트 추가, 진입 동선 설계 필요 |
| `/` 홈의 사이드 섹션 (Top N) | 노출도 높음, 재방문 유도 | 방 목록이 주인공인데 시선 분산 |
| `/me` 탭 추가 ("내 예측 성적") | `member_id` 이력 인덱스와 정확히 맞음 | 전체 랭킹을 보여줄 자리가 없음 |
| `/rooms/:id/result` 하단 카드 | 결과 확인 직후 문맥이 좋음 | 방별 데이터가 아니라 일자별 전역 데이터라 문맥이 어긋남 |

`/leaderboard`(전체) + `/me` 탭(내 이력) 조합이 인덱스 구조와 가장 잘 맞는다(제안, 결정 아님).
표시 컬럼 후보: `rank_no`, 닉네임, `score`, `prediction_count`, `rewarded_count`, `average_error_rate`, `best_error_rate`.
공동 순위가 불가능하다는 점(§3.9)에 주의.

### 5.3 그 밖에 대응 화면이 없는 것

- **`report`** — 신고 UI. 진입점 4곳(§3.10)이 필요하고, 신고 후 상태 조회 화면은 선택 사항이다.
  후보: 각 대상 화면의 컨텍스트 메뉴 + 공용 신고 모달(`shared/ui/`는 두 번째 등장 시 추출 규칙 적용).
- **`point_charge`** — 포인트 충전. `/wallet` 내 모달 또는 `/wallet/charge` 신설.
  PG 결제는 외부 리다이렉트가 필요하므로 콜백 라우트(`/wallet/charge/callback` 등)가 추가로 필요하다(추정).
- **`deal_shipping`** — 배송지 입력·송장 입력·수취 확인. `/rooms/:id/result`에 붙일지 `/me` 거래 탭에 붙일지
  선택이 필요하다. `auto_confirm_at` 카운트다운 위치도 같이 정해야 한다.
- **`product_image` 다중 업로드·정렬·대표 지정** — `/rooms/new` 폼 안. §3.4의 라디오/순서 제약 반영 필요.

## 6. 백엔드에 확인이 필요한 것

스키마만으로는 판단할 수 없는 항목이다. 우선순위 순.

### 6.1 CHECK 제약이 없어 값 목록을 알 수 없는 컬럼

가장 시급하다. 아래 컬럼들은 전부 `VARCHAR NOT NULL`인데 CHECK가 없어서 FE가 취할 수 있는 값을 알 수 없다.

| 테이블 | 컬럼 | FE 영향 |
| --- | --- | --- |
| `auction_room` | `status` VARCHAR(30) | **최우선.** `Room.state`(`live`/`waiting`/`ended`)가 이걸 매핑한다. 방 목록 필터·라우팅 분기·라이브 진입 가능 여부가 전부 여기 달림 |
| `chat_message` | `message_type` VARCHAR(20) | 일반/시스템/질문 등 렌더 분기 |
| `chat_message` | `moderation_status` VARCHAR(20) | 가림 처리 조건. `product`와 같은 4값인지 |
| `chat_message` | `question_queue_status` VARCHAR(20) | CHAT-06 질문 큐 상태 배지 |
| `deal` | `deal_status` VARCHAR(30) | DEAL-07 "지금 할 일" 계산의 핵심 |
| `deal` | `escrow_status` VARCHAR(20) | 에스크로 상태 표시 |
| `point_charge` | `status` VARCHAR(20) | 충전 내역 상태 배지 |
| `trade_point_ledger` | `transaction_type` VARCHAR(40) | 원장 각 행의 한글 라벨 매핑 |
| `betting_point_ledger` | `transaction_type` VARCHAR(40) | 위와 동일 |
| `report` | `status` VARCHAR(20) | 신고 처리 상태 |
| `report` | `reason_code` VARCHAR(40) | 신고 사유 선택 목록. FE 하드코딩 불가 |
| `auction_result` | `termination_reason` VARCHAR(30) | 유찰·취소 사유 문구 |

`termination_reason`은 `NOT NULL`이라 `SOLD`인 경우에도 값이 들어간다 — 정상 종료를 뜻하는 값이 무엇인지도 함께 확인.

### 6.2 클라이언트 책임 범위

- **`idempotency_key` 생성 주체.** `bid`·`prediction`·두 원장 모두 `NOT NULL UNIQUE`다.
  클라이언트가 UUID를 만들어 보내는가, 서버가 생성하는가. 클라이언트라면 재시도 시 같은 키를 재사용해야 하고,
  포맷 제약(길이 100자 이내, UUID v4 여부)도 확인이 필요하다.
- **`bid.sequence_no` 생성 주체.** 클라이언트가 "내 입찰 N번째"를 세어 보내는가, 서버가 매기는가.
  클라이언트라면 재연결 후 카운터 복구 문제가 생긴다.
- **입찰 최소 단위·시작가 미만 입찰 처리.** 스키마에 `amount > 0`뿐이다. 호가 단위가 있는가.
- **`auction_participant` 생성 시점.** 방 입장 시인가, 첫 입찰/예측 시인가.
  참여자 수(`participantCount`)의 의미가 달라진다.
- **역할 전환 허용 여부.** `UNIQUE(auction_room_id, member_id)` 때문에 BIDDER ↔ PREDICTOR 전환이
  UPDATE로만 가능하다. API가 이를 허용하는가, 제출 이력이 있으면 막는가. §3.1의 토글 설계가 여기 달렸다.

### 6.3 데이터 형식

- **BIGINT ID의 JSON 직렬화.** `number`로 오는가 `string`으로 오는가.
  `number`면 JS `Number.MAX_SAFE_INTEGER`(2^53) 초과 시 정밀도가 깨진다. 현재 FE는 전부 `number`다.
- **타임스탬프 포맷.** `TIMESTAMPTZ`가 ISO 8601 문자열로 오는가 epoch ms로 오는가.
  `live` 스토어의 `endsAt`은 epoch ms를 가정하고 있다.
- **`serverOffset` 획득 방법.** AGENTS.md 규칙 5가 요구하는 서버-클라이언트 시각 보정값을
  어떤 API/헤더/소켓 이벤트로 받는가. 스키마에는 근거가 없다.
- **`prediction_result.error_rate` 단위.** `NUMERIC(8,6)`이 비율(0.05 = 5%)인가 퍼센트(5.0 = 5%)인가.
- **`prediction_result.payout_amount`가 원금 포함인가 순수익인가.** `reward_multiplier = 1`이
  "원금 반환"이라면 포함으로 보이지만(추정), 확인이 필요하다. `estimatedPayout` getter가 이에 의존한다.
- **오차 밴드 경계값.** ×5/×3/×1을 가르는 오차율 임계값(1%/3%/5% 추정)이 서버 상수인가.
  FE가 "1% 이내면 ×5" 같은 안내 문구를 띄우려면 값이 필요하다.

### 6.4 API 형태

- **방 목록 API가 집계값을 주는가.** `bidCount`·`participantCount`는 컬럼이 아니라 집계다.
  목록 응답에 포함되는가, 별도 호출인가.
- **썸네일 URL 제공 방식.** `product_image` 배열 전체인가 썸네일 URL 하나인가.
  대표 이미지가 0장인 경우 서버가 폴백을 주는가.
- **방 개설이 단일 요청인가.** `product` + `product_image[]` + `auction_room` 3개 테이블을
  한 번에 만드는가, 단계별인가. 이미지 업로드는 별도 엔드포인트인가.
- **카테고리 목록 API.** `product_category`를 언제 받아 캐시하는가.
- **결과 화면의 입찰 내역 공개 범위.** `second_highest_bid_amount`를 화면에 노출하는가.
  전체 입찰 내역을 마스킹해서 준다면 마스킹 규칙은 무엇인가 (AGENTS.md 규칙 6).

### 6.5 정책·범위

- **`moderation_status = 'BLOCKED'` 상품·이미지 처리.** 목록에서 제외되는가, 가림 처리되는가.
  판매자 본인에게는 보이는가.
- **판매자 본인의 방 참여.** `deal`에 `seller <> buyer` 제약이 있어 입찰은 불가하다.
  예측 베팅은 가능한가.
- **`member.status = 'SUSPENDED'` 회원의 접근 범위.** 조회만 가능한가, 로그인 자체가 막히는가.
- **`member.role = 'ADMIN'` 관리자 화면이 이번 MVP 범위인가.**
  `report` 처리·`chat_message` 검수·`product` 승인이 스키마상 전제되어 있다.
- **`prediction_leaderboard` 갱신 주기.** `ranking_date`가 DATE이므로 일 단위 배치로 보인다(추정).
  당일 데이터는 언제 생기는가 — 실시간 랭킹은 불가능한 구조로 보인다.
- **약관 동의 철회의 범위.** 필수 약관도 철회 가능한가(=탈퇴 처리인가), 선택 약관만인가.
